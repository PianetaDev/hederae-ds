02 — Figma Organization

Come organizzare i file Figma

La struttura del file Figma cambia radicalmente in base al tier. Più è alto il tier, più i file devono essere separati, referenziati e versionati come se fossero un prodotto software.

Tier 3–4 — Un file, tutto dentro

Single File Structure

Per progetti semplici (Rara, Corner Table). Un unico file Figma, organizzato con pagine dedicate. Gestibile da 1-2 designer.
📁 [Brand] — Design 📄 Cover ← branding del file 📄 🎨 Styles ← color/text styles, variables 📄 🧩 Components ← tutti i componenti 📄 📐 Wireframes ← layout a bassa fedeltà 📄 🖥️ Desktop ← visual design finale 📄 📱 Mobile ← responsive views 📄 🔍 Prototype ← flussi interattivi 📄 📦 Archive ← versioni precedenti
  • Ogni componente nel file deve essere un Figma component (non gruppo)
  • I Figma Variables devono rispecchiare esattamente i token JSON del DS
  • Le pagine Archive non si cancellano — si spostano in fondo e si rinominano con data
  • Nessun colore esplicito nei frame — sempre via styles o variables
Tier 2 — Library + Products

Multi-File System

Per ecosistemi medi (Susdef). File separato per DS Library (pubblicato come shared library) e un file per prodotto.
📚 [Brand] — DS Library ← PUBLISH 📄 🎨 Tokens 📄 🧩 Components 📄 📖 Documentation 🖥️ [Brand] — Site A 📄 🗺️ IA & Flows 📄 📐 Wireframes 📄 🎨 Visual 📄 🔍 Prototype 🖥️ [Brand] — Site B 📄 ...
  • Il DS Library è un file pubblicato come library e mai editato direttamente nei file prodotto
  • I brand token overrides (colori, font) vivono come Figma Variables locali nel file prodotto
  • Ogni modifica al DS Library richiede notifica al team prima del publish
Tier 1 — Enterprise Ecosystem (AGESCI)

Team Library + Multi-Product Architecture

Per ecosistemi istituzionali. Figma Team dedicato, con DS condiviso come Team Library. Ogni prodotto (nazionale, regionale, etc.) ha file separati per fase (UX / Visual / Handoff).
🏛️ Figma Team: [Brand] Ecosystem │ ├── 📚 Hederae Core Library ← TEAM PUBLISH (condivisa da Pianeta Studio) │ 📄 Primitives spacing, radius, motion, palette │ 📄 Semantic Tokens surface, content, interactive │ 📄 Base Components atoms: button, input, badge, icon... │ ├── 🎨 [Brand] — Brand Library ← TEAM PUBLISH (brand-specific) │ 📄 Token Overrides variabili brand: colori, font, radius │ 📄 Brand Components molecole branded: navbar, footer, hero │ 📄 Iconography icon set del brand │ 📄 Illustration illustrazioni, assets grafici │ ├── 📐 [Brand] — UX Architecture ← working file (non pubblicato) │ 📄 Information Architecture sitemap, gerarchia pagine │ 📄 User Flows flussi per persona (nazionale, regionale...) │ 📄 Wireframes Desktop │ 📄 Wireframes Mobile │ ├── 🖥️ [Brand] — Visual Design ← working file │ 📄 Homepage Nazionale │ 📄 Homepage Regionale │ 📄 Sottosito Branca │ 📄 Pagina Notizia │ 📄 [pagina per tipo]... │ └── 📊 [Brand] — Dev Handoff ← handoff file 📄 Component Specs annotazioni, stati, responsive 📄 Token Reference mappa token → CSS var 📄 Accessibility Notes checklist WCAG per componente
  • Mai editare Hederae Core Library direttamente — modifiche proposte via PR nel repo GitHub hedera-design-system
  • I file UX e Visual sono separati — il UX può avanzare mentre il visual è in review
  • Ogni component nel Brand Library deve avere un Figma component story con tutti gli stati (default, hover, disabled, focus, error)
  • Il Dev Handoff file è generato da Visual Design, mai editato manualmente
  • Nessun "one-off design" — se un pattern esiste più di 2 volte, diventa componente
  • Ogni pagina di Visual Design usa auto layout su tutti i frame (required per handoff accurato)

Il differenziatore: 1 route = 1 sito satellite

Con Terra (Payload CMS) + Hederae multi-site: aggiungere AGESCI Piemonte o una nuova zona è configurare una route e assegnare un brand config — non costruire un nuovo sito. Questo è il valore competitivo principale da comunicare nella gara.
